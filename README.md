# CivIQ — AI Decision Intelligence for Smarter Communities

Built for: **AI for Better Living and Smarter Communities** (Gen AI APAC hackathon)
Solution areas covered: urban mobility, environmental sustainability, energy efficiency, citizen engagement

## What it does

CivIQ is a decision-intelligence dashboard for city stakeholders (here, modeled on Hyderabad wards). It ingests civic data across five domains — air quality, transit, energy load, and citizen complaints — and turns it into:

1. **Live KPI monitoring** — at-a-glance ward health, styled as stamped municipal forms.
2. **AI anomaly detection** — flags statistically unusual patterns (e.g. an AQI spike correlated with construction + low wind, a transit corridor with a sustained bottleneck) rather than raw threshold breaches.
3. **Ranked recommendations** — concrete, costed actions an official could action this week, not generic advice.
4. **Natural-language query desk ("Ask CivIQ")** — officials or citizens ask plain-English questions ("Which zone has the worst air quality?") and get answers grounded in the live dataset, with numbers cited, not hallucinated.

## Why it fits the brief

- **Conversational analytics / NL interface** — the "Ask CivIQ" panel is a constrained natural-language interface over structured city data.
- **LLM + RAG** — answers are generated by retrieving the relevant slice of ward data and grounding the model's response in it (see `askGemini()` in `app.js`), so the model can't invent figures.
- **Predictive analytics** — the energy panel forecasts grid load 24h ahead and flags capacity risk before it happens, not after.
- **Anomaly detection & automated workflows** — the anomaly engine surfaces correlated, non-obvious patterns (e.g. AQI spike + low wind + active construction) the way a human analyst would, and recommendations are generated from those anomalies rather than hardcoded.
- **Responsible & explainable AI** — every anomaly and recommendation cites the specific numbers and time window behind it, so a city official can verify the claim rather than trust a black box.

## Architecture

```
Citizen / city data sources                  This prototype
 ── air quality sensors      ┐                ┐
 ── transit GPS/telemetry    ┤   →  BigQuery  │  → static dataset (data.js)
 ── utility/grid telemetry   ┤   (data layer) │     simulating the BigQuery views
 ── citizen complaint portal ┘                ┘

         ↓ aggregated views

   Looker / Vertex AI Agent Builder           CivIQ dashboard (index.html)
   ── anomaly detection models      →         ── KPI + chart rendering (Chart.js)
   ── Gemini (via Vertex AI) for NL  →        ── anomaly + recommendation panels
      grounded Q&A and recommendation             generated from the same data layer
      generation, using RAG over the live      ── "Ask CivIQ" — NL query box
      BigQuery views as context
```

The prototype runs entirely client-side with a synthetic dataset (`data.js`) so it's demoable without cloud credentials. The "Ask CivIQ" box has two modes, both implemented in `app.js`:

- **Default (offline-safe):** a deterministic retrieval+template engine (`localGroundedAnswer()`) that reasons over the same dataset the charts use — this is what runs out of the box.
- **Live mode:** if you paste a Gemini API key into `GEMINI_API_KEY` at the top of `app.js`, queries route through `askGemini()`, which sends the full ward dataset as grounding context to `gemini-2.0-flash` and returns a cited, data-grounded answer. In a full Google Cloud deployment this call would instead go through **Vertex AI Agent Builder / Agent Development Kit (ADK)**, with the data layer served by **BigQuery** and **AlloyDB**, anomaly/forecast models run via **Vertex AI** pipelines, and the dashboard layer optionally backed by **Looker** for the official analyst view.

## Files

- `index.html` — page structure and styling (municipal-ledger visual identity: stamped cards, ticket numbers, ledger-paper background — distinct from a generic dark ops dashboard, since the subject is civic record-keeping)
- `data.js` — synthetic Hyderabad ward dataset (AQI, energy forecast, transit delays, complaint volumes) standing in for BigQuery views
- `app.js` — chart rendering, anomaly detection logic, recommendation generation, and the NL query engine

## Running it

Just open `index.html` in a browser — no build step, no server required. To enable live Gemini-backed answers, add a Vertex AI / Gemini API key in `app.js` (`GEMINI_API_KEY`).

## Extending for a real deployment

- Replace `data.js` with BigQuery query results (the shape is already designed to mirror what BigQuery views would return).
- Swap the anomaly rules in `data.js`/`app.js` for a Vertex AI anomaly-detection model output.
- Move the NL query handling to a Vertex AI Agent Builder agent with tool access to BigQuery, so it can answer open-ended questions beyond the four templated categories.
- Add Cloud Run for hosting and Cloud Functions for the data-ingestion pipeline from city sensor APIs.
